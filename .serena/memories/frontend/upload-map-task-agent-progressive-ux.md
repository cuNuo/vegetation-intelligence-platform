# 上传、地图、任务与 Agent 渐进式体验优化

## 一句话结论

- 地图只允许导入完成首次定位和用户点击定位改变视角，定位 zoom 上限为 16；计算必须使用原始 GeoTIFF；地图主视图保留颜色条、比例尺和指北针；Agent 方案卡片有完整边界和常驻执行单；思考过程逐阶段展示；任务管理器支持导出 TIF；知识库必须避免测试数据污染和无关具体病害召回。

## 稳定约束 / 正确做法

- `syncMapLayers()` 只做图层同步，不允许调用 `fitBounds()`；定位 zoom 上限为 16。
- 计算链路必须通过 Rasterio window 读取原始 GeoTIFF；PNG/缩略图不得参与计算。
- 地图主视图必须显示结果颜色条、可读比例尺和不重叠的指北针。
- 有 TIF 瓦片时优先使用 TIF，不要同时叠加 PNG；瓦片拉伸按影像级统一 2/98 百分位。
- 公式卡片长公式使用横向公式视窗，并保留原始公式行。
- 内置知识召回内容应包含公式、波段、范围、用途、场景和限制。
- 常见农情知识至少覆盖：通用长势、叶绿素/氮素、水分/干旱、积水/涝害、稀疏植被/裸土、盐碱、倒伏、病虫害异常边界、多时相变化。
- 遥感 Agent 只能定位疑似异常，不能在用户未明确提及具体病名时引入根腐病、白粉病、锈病等具体诊断。持久知识库中包含具体病名的文档，只能在用户问题同样包含该病名时召回。
- 测试导入的知识必须使用明确测试 source，并在测试前后清理，禁止污染真实 PostgreSQL 知识库。
- Agent 思考过程首步约 420ms，后续阶段约 1180ms；done 先以 running 显示并停留约 760ms 后完成。
- 方案摘要、执行指数、指数勾选、引擎、block size、优先级、提交按钮必须始终可见。若 `selectedIndices` 为空但存在可执行推荐，默认勾选全部可执行指数。
- 执行指数区域应与方案卡片/指标区左右边界对齐，不要出现额外缩进。
- 任务管理器完成任务后应为每个产品提供“导出TIF”。
- CI：后端 Python 3.11 `ruff + pytest`，前端 Node 22 `npm ci + npm run build`。

## 常见误区

- 不要把 `fitBounds()` 放回 watcher。
- 不要让 PNG 与 TIF 同时叠在地图主视图。
- 不要每瓦片单独拉伸。
- 不要把执行指数和提交按钮放进 `isDetailsOpen`。
- 不要一次性渲染全部 thought。
- 不要让 pytest 知识样例永久写入真实知识库。
- 不要根据只匹配“水分”等泛化词的 RAG 文档推断用户未提及的具体病害。
