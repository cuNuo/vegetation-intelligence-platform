# 上传、地图、任务与 Agent 渐进式体验优化

## 一句话结论

- 上传体验采用分阶段状态，但上传栏必须保持单行高信号状态；地图只允许导入完成首次定位和用户点击定位改变视角，定位 zoom 上限为 16；计算必须使用原始 GeoTIFF window 读取原图波段；地图显示有 TIF 时优先用原图/结果 TIF 动态瓦片，PNG 缩略图只作为无 TIF 时兜底；地图主视图必须保留颜色条、比例尺和指北针；TIF 瓦片使用影像级统一拉伸统计；Agent 方案卡片需要完整边界和常驻执行单；思考过程用前端队列逐条展示高层 SSE，done 事件先以 running 展示并停留后再完成；任务管理器完成结果必须支持导出 TIF；公式卡片长公式应使用公式视窗和原始公式行，不要硬挤换行。

## 稳定约束 / 正确做法

- 地图同步图层和移动视角必须分离；`syncMapLayers()` 只做图层同步，不允许调用 `fitBounds()`。
- 定位 zoom 上限固定从 `DEFAULT_LOCATE_MAX_ZOOM = 16` 起步，小 footprint 应定位到 16。
- 地图主视图必须显示基础制图要素：结果颜色条、比例尺、指北针。比例尺要足够可读，指北针不能与缩放控件、图层面板或颜色条重叠。
- 计算链路必须使用原始 GeoTIFF：`raster_pipeline.py` 通过 Rasterio window 读取原图波段并输出 GeoTIFF/COG；PNG/缩略图不得参与植被指数计算。
- 地图展示有 `objectKey`/TIF 瓦片时必须优先使用 TIF raster source，不要同时叠加 PNG 缩略图；PNG 预览只在没有 TIF 瓦片源时兜底。
- TIF 瓦片颜色拉伸应按影像级缓存统一 2/98 百分位范围，不要每个瓦片单独计算百分位。
- 公式卡片必须优先保证可读与可核对：可拆最外层分式时显示分式；长公式使用横向公式视窗，不要强行多行挤压；保留原始公式等宽行便于核对。
- 内置知识库召回内容应包含公式、必需波段、期望范围、用途、推荐场景和限制；Agent 不应只依赖简短描述判断指数。
- Agent 方案卡片应有完整边框、内边距和背景隔离，不要只靠顶边分割导致右侧栏割裂。
- 详情只放解释、运行 trace、来源和质量提示；方案摘要、可执行指数、指数勾选、引擎、block size、优先级、确认提交按钮必须始终可见。
- 前端应对 thought/status 做节流队列展示，避免后端快速返回时所有步骤瞬间蹦出；当前首步约 420ms，后续阶段约 1180ms。done 事件先以 running 显示，至少停留约 760ms 后再转为 done。
- thinking running 阶段应有明确视觉状态，例如脉冲点和“进行中”标识；不要让所有阶段同时以完成态出现。
- 任务管理器完成任务后应在每个产品旁提供“导出TIF”，链接到对应产品的 GeoTIFF artifact。
- 远程仓库同步后应有轻量 CI：后端 Python 3.11 `ruff + pytest`，前端 Node 22 `npm ci + npm run build`。

## 常见误区

- 把 `fitBounds()` 放回 watcher 或 `syncMapLayers()` 会重新引入抢视角问题。
- 将定位上限改回 14/13 会违背当前 UX 约束。
- 不要让 PNG 缩略图和 TIF 瓦片同时叠在地图主视图上。
- 不要用每块瓦片自己的 percentile 拉伸显示 TIF。
- 不要把完整 `plan.trace` 同时显示在思考过程和运行过程里。
- 不要把执行指数和提交按钮放进 `isDetailsOpen` 分支。
- 不要让公式文本在窄卡片里硬折行到难以理解。
- 不要把后端快速返回的 thought 直接一次性渲染完；需要前端节流和 running/done 两段式过渡。
