## 处理过程

### 1. 后端进度与瓦片缓存

- 做了什么：`JobRecord` 增加 `started_at / finished_at / eta_seconds / throughput / current / total / engine / index_count`；本地任务和 Celery 进度都写入当前窗口、总窗口、速率和 ETA；`render_geotiff_tile()` 增加 512 项内存 LRU，缓存 PNG bytes。
- 为什么这么做：任务管理器需要稳定字段；缓存 PNG bytes 比缓存 Rasterio dataset 句柄更少文件句柄风险。
- 结果：异步任务 API 保持向后兼容，新增字段可直接供前端展示。

### 2. 上传、地图和多指数联动

- 做了什么：上传栏增加“上传中 -> 后端接收完成 -> 读取元数据 -> 生成预览/瓦片就绪 -> 已定位”阶段；地图分离 PNG/范围框占位层和 TIF 瓦片层；TIF 源只在视野与影像范围相交时挂载；移除每次同步强制 `fitBounds(maxZoom=14)`；图层面板增加自动定位状态和多指数切换。
- 为什么这么做：100% 后仍在后端解析时要给用户明确反馈；地图不应在用户手动拖拽/缩放后抢视角；指数切换必须同时驱动地图、统计和任务管理器。
- 结果：首次上传和首次打开结果会自动定位一次，随后自动定位关闭；用户点击定位时仍可主动回到目标范围。

### 3. Agent 与任务管理器

- 做了什么：Agent 改成固定头部、滚动中段、固定底部输入；方案详情和执行单进入中段滚动区；轮询 job 事件不再反复追加到聊天消息；“任务脉冲”改为“任务管理器”，展示状态、进度、ETA、速率、已用时间、引擎、指数数量、窗口。
- 为什么这么做：对话区应保留解释和建议，执行状态主入口应回到任务管理器；输入框永远贴底能减少等待时的视角打断。
- 结果：Agent 输入区不再被方案详情挤走，任务面板成为作业状态主视图。

### 4. 二次 UX 修正

- 做了什么：按用户截图反馈取消结果首次打开自动定位，只保留导入完成首次定位与用户点击定位；TIF raster source 改为常驻并写入 `bounds`，不再在 `moveend` 反复挂载/卸载；导入栏恢复足够高度并把阶段反馈压缩为单行；Agent 增加 `thought` SSE 事件、思考过程面板、来源去重与折叠展示。
- 为什么这么做：图层切换和结果切换不应改变用户当前缩放等级；过度压缩工具栏会遮蔽队列和金字塔内容；重复来源和长质量提示会挤占对话空间。
- 结果：除导入首次定位和定位按钮外，地图只管理图层不改 zoom/center；Agent 的状态流进入紧凑思考过程，不再刷屏。

### 5. Agent 空态重排

- 做了什么：将“待命 / 规则引擎兜底 / 网络检索开启 / NEW SESSION”从对话中部移到标题下方的细状态栏，移除中间大状态卡和两个方块 chip。
- 为什么这么做：这些信息属于面板状态，不应打断聊天阅读路径或占据对话主体空间。
- 结果：Agent 空态只保留欢迎语和输入框，状态信息更贴近标题且不遮挡消息流。

### 6. 底部遮蔽与思考过程重叠修正

- 做了什么：删除上传完成后的“已按影像范围定位，TIF瓦片将按视野渐进加载”提示；上传栏不再渲染底部详情行，上传中把阶段、进度和文件名合并到单行；Agent 将“思考过程”从对话块内拆出为独立可滚动区，并限制最大高度；后端 `thought` SSE 补充影像尺寸、可用波段、外部资料数量、推荐指数、可执行数量、引擎和估算内存。
- 为什么这么做：上传栏底部小字会在窄高工具栏内被裁切；思考过程和方案详情处于同一滚动层时会互相遮挡；用户需要看到具体推理上下文而不是重复的泛化步骤。
- 结果：上传栏只保留高信号状态，不再出现底部遮蔽；右侧 Agent 的对话、思考过程、方案详情按块顺序滚动，不互相重叠，且思考内容更具体。

### 7. 对话栏最终重排与定位 zoom 上限

- 做了什么：将“收起/展开详情”按钮移入方案标题栏右侧，避免卡在思考过程附近；消息列表和思考过程取消内层滚动，只保留 Agent 中段一个外层滚动；知识库/配置按钮增加最小宽度并禁止逐字换行；地图定位自适应上限改为 `DEFAULT_LOCATE_MAX_ZOOM = 16`，小 footprint 定位到 16。
- 为什么这么做：窄侧栏内多层滚动条和跨区按钮会造成视觉遮蔽与误读；用户明确要求默认缩放等级不要 14，改为 16。
- 结果：Agent 右侧栏形成“头部状态 / 单一滚动内容 / 固定输入”三段结构，思考、方案和输入不再互相挤压；地图仍只在导入完成首次定位或用户点击定位时移动视角，但目标 zoom 上限调整为 16。

### 8. Agent 重复 trace 与影像边缘色差修正

- 做了什么：Agent 思考过程不再合并完整 `activePlan.trace`，后端 SSE 也不再把完整 trace 重复推送到思考区；完整运行过程改成方案详情内折叠项，方案详情默认收起；确认任务的 `status/done` 事件只更新执行摘要，不再追加聊天消息。地图展示改为有 TIF 时优先使用原图/结果 TIF 瓦片，不再叠加 PNG 缩略图；后端瓦片渲染为同一影像缓存统一 2/98 百分位拉伸范围，并把有效像元 alpha 改为 255。
- 为什么这么做：重复 trace 会让思考区和方案区堆叠、错位；PNG 缩略图与 TIF 瓦片同时叠加会造成边缘色差和范围错觉；每块瓦片单独拉伸会导致相邻瓦片/边缘亮度不一致。计算链路确认仍在 `raster_pipeline.py` 中按原始 GeoTIFF window 读取原图波段计算，不使用缩略图。
- 结果：右侧 Agent 只显示高层流式思考，细节保留但默认折叠；地图视觉优先来自原图/结果 TIF 瓦片，缩略图只在没有 TIF 时兜底；瓦片色彩跨边界更一致。

### 9. 执行单外置、思考节流与 TIF 导出

- 做了什么：Agent 方案卡片在详情收起时仍显示可执行指数、指数勾选、引擎、block size、优先级和确认提交按钮；思考过程增加前端队列，以约 360ms 间隔逐条显示 SSE thought/status；思考区加最大高度与内部滚动边界，方案卡片加背景隔离，避免重叠；任务管理器在每个完成产品旁增加“导出TIF”链接，指向产品 GeoTIFF artifact。
- 为什么这么做：用户收起详情后仍需要知道将执行哪些指数并能提交任务；后端响应快时所有 thought 会瞬间出现，缺少“正在思考”的阶段感；完成任务应能直接下载 TIF 格式结果。
- 结果：详情收起不再隐藏主操作；思考过程逐步出现；任务完成后可以从任务管理器直接导出每个指数对应的 TIF。
