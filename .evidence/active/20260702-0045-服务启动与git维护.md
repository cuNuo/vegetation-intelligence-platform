# 服务启动与 git 维护

## 背景

- 用户反馈当前无法启动服务，且仓库存在大量未提交内容。
- 当前仓库是无初始提交状态，`main` 分支上所有项目文件均为未跟踪文件。
- 假设本轮目标是恢复本地前后端可访问状态，并整理 git 忽略边界，避免本地生成物进入提交。

## 诊断

- 后端启动命令：
  `D:\miniconda\envs\giskeshe\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8011`
- 复现结果：FastAPI startup 已完成，但绑定 `127.0.0.1:8011` 失败，错误为端口已被占用。
- 端口占用来源：旧的 `uvicorn app.main:app --reload` Python 进程。
- 前端 `npm run build` 通过，只有 Vite chunk 体积警告。

## 处理

- 重启后端服务，日志写入：
  - `.evidence/runtime-logs/backend-8011-20260702-restart.log`
  - `.evidence/runtime-logs/backend-8011-20260702-restart.err.log`
- 启动前端开发服务，日志写入：
  - `.evidence/runtime-logs/frontend-5174-20260702-restart.log`
  - `.evidence/runtime-logs/frontend-5174-20260702-restart.err.log`
- 更新 `.gitignore`，忽略本地运行产物：
  - `%SystemDrive%/`
  - `texput.log`
  - `data/`
  - `output/`
  - `.playwright-mcp/`
  - `backend/*.egg-info/`
- 同步 `frontend/vite.config.ts` 的开发端口为 `5174`，避免与实际生效的 `vite.config.js` 端口不一致。
- 移除前端硬编码天地图 Token，改为通过 `VITE_TIANDITU_TOKEN` 从部署环境注入；`.env.example` 仅保留空占位。
- 为前端应用 TypeScript 配置补充 `vite/client` 类型，保证 `import.meta.env` 通过类型检查。

## 验证

- `http://127.0.0.1:8011/api/system/capabilities` 返回 200，能力信息包含 `cuda: true`、`indexCount: 30`、`asyncJobs: true`。
- `http://127.0.0.1:5174/` 返回 200。
- 后端质量检查：`ruff check .` 通过。
- 后端测试：`pytest -q` 结果为 29 passed，存在 1 个 Starlette TestClient 上游弃用警告。
- 前端构建：`npm run build` 通过，存在 Vite 大 chunk 警告。
- 敏感信息扫描发现历史天地图 Token 已移除；保留的 MinIO 示例口令仅为本地 compose/.env 示例值。

## 风险与后续

- 仓库当前仍没有任何提交，建议建立一次明确的初始提交基线。
- `.evidence/runtime-logs/` 中历史日志较多，若希望仓库更轻，可后续约定只提交关键任务日志，运行日志按需归档。
- 前端同时存在 `vite.config.js` 与 `vite.config.ts`，当前实际以 `vite.config.js` 为准；后续可考虑删除一个配置文件以消除歧义。
