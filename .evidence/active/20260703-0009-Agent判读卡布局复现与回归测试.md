## 结论

- 该问题的核心触发点是固定高侧栏中纵向 flex item 默认允许 shrink，导致 `.plan-card` 盒高被压扁，而内部执行单继续外溢，后续判读卡按错误位置排布。
- `nextActions` 等列表字段的运行期形状漂移是另一类布局放大风险，本次仍保留后端规整、前端 API 边界规整、组件展示兜底、单测复现和真实浏览器几何检查五层防线。

## 根因分析

1. 固定高 Agent 面板中，`.agent-scroll` 使用纵向 flex 布局；flex item 默认 `flex-shrink: 1`。
2. 当方案卡内容很长时，浏览器会把 `.plan-card` 的实际盒高压缩到几十像素；但 `.execution-sheet` 等子内容仍然按自身高度绘制并外溢。
3. `.insight-card` 作为下一个 flex item，会按被压缩后的 `.plan-card` 盒子位置排版，于是视觉上盖到执行单区域。
4. 前端原先还假设 `AgentResultInterpretation.nextActions` 恒为 `string[]`，外部 LLM 或兼容接口返回字符串时会被 Vue `v-for` 按字符迭代，这是另一条会放大高度的风险。
5. 原判读意见区域早期样式只靠分隔线承接，缺少独立卡片边界；当内容异常膨胀或 flex item 被压缩时，用户看到的是执行单、判读卡和下一步建议混在一起。
